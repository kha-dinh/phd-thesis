\section{Background and Related work}
\label{sec:background}

\subsection{AMD SEV} \label{sec:bg:amdsev}
\gls{sev} isolates the execution of \glspl{cvm} from the untrusted hypervisor through transparent memory encryption and access control.
% Memory encryption is performed by the \gls{sp}, using a \gls{vek} uniquely provisioned to each \gls{cvm}.
\gls{sevsnp} is the newest iteration of SEV that encrypts register states in \gls{vmsa} and addresses mapping integrity through a hardware-maintained \gls{rmp}. 
% \textcolor{red}{Explain, VMCB: for our exit code checking , RMP: to explain in advance that hypervisor relocating stuff is not possible}

\hdr{VMSA update on VMExits}
When a VMExit transitions from \gls{cvm} to host execution, the processor automatically updates the \gls{vmsa} with exit information.
Specifically,  the \texttt{vmsa.exit\_code} field is set as the exit reason for the current VMexit, e.g., 0x400 for \gls{npf}.
This recording is unavoidable as it takes place at the hardware level without the intervention of the hypervisor~\cite{amd64} and is non-forgeable as the \gls{vmsa} must be mapped as a private \gls{cvm} page that is automatically encrypted.

% Our examination of the AMD SEV-SNP specification~\cite{amd64} and the KVM implementation confirms that this

\hdr{Mapping integrity with RMP}
\gls{sevsnp} provides mapping integrity through the \gls{rmp}, which contains integrity-protected mapping information such as the page's state (encrypted or non-encrypted) and mapping sizes.
On any mapping changes, the mapping information in the \gls{rmp} is invalidated; the \gls{cvm} must \emph{validate} this update with the \texttt{pvalidate} instruction.
The \gls{rmp} is consulted by the CPU on every memory access, which allows a \gls{cvm} to detect illegal changes to the mappings.

\subsection{Side-channel threats in CVMs}
\label{sec:background:side-channels}
Side-channel attacks that undermine the confidentiality of trusted executions have long plagued Intel SGX, and many such threats continue in CVMs.


\hdr{NPF Controlled-channel attacks}
\label{sec:background:controlled-channel}
Adversaries who control the untrusted hypervisor can leverage its control over the \gls{npt} to strategically trigger \gls{npf} on \gls{cvm} pages by marking a page \emph{non-present}.
When the victim \gls{cvm} accesses such pages, the CPU generates a \texttt{\#NPF}, with the corresponding faulting address recorded in the \gls{vmcb}.
In the current SEV-SNP~\cite{amd64}, the processor notifies the hypervisor of (1) the faulting \gls{gpfn} (e.g.,\texttt{ 0xff...ab\textbf{000}}) through {\tt VMCB.EXITINFO2} and (2) the cause of the fault, along with other information~\cite{amd64} through the {\tt VMCB.EXITCODE} upon the \gls{npf} occurrence.
These so-called \gls{npf} controlled-channel attacks are adaptations of the page fault-based attacks against Intel SGX~\cite{xu2015controlledchannel,hahnel2017highresolution}.
Through the attack, the adversary can trace instruction fetches and data accesses at page granularity.
In many previous SEV(-SNP) attacks, NPF attacks have been employed as a CVM memory profiling primitive for pinpointing attack targets (\cref{sec:relatedwork:profiling}).

% We classify this attack methodology as an \emph{\gls{npf} controlled-channel attack}, drawing parallels to similar controlled-channel attacks previously documented on 


\hdr{Fine-grained side-channels}
\label{sec:background:fine-grained-sc}
Fine-grained side-channel attacks represent attack vectors that leak memory access patterns at a resolution finer than the page size.
These attacks typically exploit microarchitectural leakages (e.g., cache~\cite{sgx-cache-attack,cache-attack-sgx,hahnel2017highresolution} and TLB~\cite{wang2017leaky}) or architectural leakages such the encrypted memory access patterns~\cite{leeoffchip}, and ciphertext updates~\cite{li2021cipherleaks,li2022systematic}.
Fine-grained side-channel attack vectors have been extensively studied in the context of Intel SGX~\cite{sgx-cache-attack,cache-attack-sgx,sgxstep,bulck2018nemesis,hahnel2017highresolution,cachezoom,sgxspectre,foreshadow}, and certain attacks remain feasible against SEV-SNP \glspl{cvm}.
For instance, recent research highlights the exploitation of L2 cache Prime+Probe and ciphertext side-channel attacks to extract sensitive information from SEV CVMs~\cite{li2021cipherleaks,li2022systematic,sevstep}.
Ciphertext side-channel attacks~\cite{li2021cipherleaks,li2022systematic,ciphersteal} are specific to AMD SEV(-SNP).
They exploit the weak AES-XEX cipher used for CVM memory encryption, which consistently produces the same ciphertext for identical plaintext written to the same physical address.
In combination with other primitives, a previous work~\cite{li2021cipherleaks} demonstrated a case of full RSA key leak.

\subsubsection{Untrustworthy interrupts and temporal resolution}
\label{sec:background:interrupts}
% \textcolor{red}{Need details}
A SEV-SNP CVM currently has no option but to rely solely on the untrusted hypervisor to deliver \emph{virtualized} interrupts, which may be delayed or completely skipped.
Also, the hypervisor may leverage the precise hardware interrupt timer (e.g., APIC) to seize the victim's CVM execution for itself at any moment and at an arbitrary frequency.
This untrustworthy interrupt delivery relation has been exploited as a primitive to boost the temporal resolution of the fine-grained side channels.
Since the fine-grained side channels must target very specific addresses and are often noisy due to the ongoing CVM computations, \emph{single-stepping} technique~\cite{sevstep,sgxstep,cachewarp} was often used to acquire temporal resolution of a single instruction.

T-SGX retrofits the Intel-specific TSX feature to detect page faults during sensitive computation.

SGX-LAPD proactively accesses memory pages before use and checks for enclave exits.

Instead of forbidding all enclave exits, Varys~\cite{oleksenko2018varys} advocated periodically sampling the enclave's exit rate and terminating based on a set threshold.
