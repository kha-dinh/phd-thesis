Modern hardware increasingly embeds security primitives that, if properly integrated, can enforce fine-grained protection within software.
Yet, mainstream \glspl{os} continue to rely on process-based protection, introducing \emph{protection incoherencies}: while hardware features such as Intel PKU, ARM PA and MTE provides security guarantee in the userspace, the OS continues to mediate access at a coarse-grained, process-level.

This chapter presents \capacity, a hardware-assisted intra-process protection framework that reconciles these incoherencies by retrofitting a hardware-assisted capability-based protection into existing OS abstractions.
By transforming memory and file references into cryptographically verifiable capabilities, \capacity extends trust enforcement beyond the kernel boundary and establishes hardware-backed integrity across in-process domains.

In the following, I first explain how \capacity composes within the framework of coherent protection established in~\cref{chap:intro}.
Then, the remainder of the chapter details the design, implementation, and evaluation of \capacity, demonstrating how principled retrofitting enables efficient and coherent in-process isolation on modern ARM systems.

\section{Toward Coherent In-process Isolation}

Despite the promise of fine-grained security and low performance overheads, modern hardware mechanisms do not integrate cleanly with existing \gls{os} abstractions.
The reason is that most focus primarily on memory isolation, whereas the \gls{os} enforces access control across all resource types (e.g., files and sockets).

\hdr{Kernel as a confused deputy}

The seminar paper by Connor et al. ~\cite{connor2020pku} demonstrates the perils of this problem, highlighting the risk of \emph{confused-deputy attacks}~\cite{hardy_confused_1988} targeting the kernel~\cite{pku-pitfall}.

For example, on PKU-based systems, the \gls{os} is unaware of an application's internal protection domains and continues to enforce access control only at process granularity.
A domain, for instance, may use the \texttt{mmap} system call to \emph{remove} \gls{pku} protection on a target page.
Even worse, some system calls such as \texttt{process\_vm\_readv} and \texttt{send} read from userspace memory without regard for in-process domains, enabling one domain to bypass memory isolation~\cite{pku-pitfall,yang2025endokernel}

\hdr{Incoherency in protection}

I posit that the root cause for the confused deputy problem, is the \emph{incoherency of protection}, introduced when when trying to retrofit a hardware-assisted protection model into the existing process-based one established by the OS.

\hdr{Capacity's approach}

Observing that features of PA and MTE on moder ARM processors exhibit characteristics of capability-based security, \capacity propose a capability-based protection scheme that coherently protect both userspace references (i.e., pointers), and references to the systems objects.
This approach enable protection of sensitive resources throughout their life-cycles.

\capacity enjoy hardware acceleration for reference authentication to achieve reasonably low overheads, and at the same time, cryptographically-authenticated reference mechanism significantly simplifying the enforcement mechanism (KISS principle~\cite{kiss}); both userspace and system resource references are authenticated with a single instructions.

\section{Abstract}
In-process compartmentalization and access control have been actively explored to provide in-place and efficient isolation of in-process security domains. Many works have proposed compartmentalization schemes that leverage hardware features.
Newer ARM architectures introduce Pointer Authentication (PA) and Memory Tagging Extension (MTE), adapting the reference validation model for memory safety and runtime exploit mitigation.
Despite their potential, these features are underexplored in the context of userspace program compartmentalization.

This paper presents \thename, a novel hardware-assisted intra-process access control design that embraces capability-based security principles.
\thename coherently incorporates the new hardware security features on ARM, based on the insight that the features already exhibit inherent capability characteristics.
It supports the life-cycle protection of the domain's sensitive objects -- starting from their import from the file system to their place in memory.
With intra-process domains authenticated with unique PA keys, \thename transforms file descriptors and memory pointers into cryptographically-authenticated references and completely mediates reference usage with its program instrumentation framework and an efficient system call monitor.
We evaluate our \thename-enabled NGINX web server prototype and other common applications in which sensitive resources are isolated into different domains. Our evaluation shows that \thename incurs a low-performance overhead of approximately $17\%$ for the single-threaded and $13.54\%$ for the multi-threaded webserver.
