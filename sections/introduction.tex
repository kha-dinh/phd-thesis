\chapter{Introduction}
\pagenumbering{arabic}
\label{sec:intro}

The security primitives provided by \glspl{os} lay the foundation of many modern software systems. For instance, \gls{os} process abstraction not only enables efficient multitasking on a processor but also provides a unit of isolation to perform \emph{privilege separation}. Many real-world programs such as the Chrome web browser~\cite{chrome-security} and OpenSSH~\cite{openssh} choose to split programs into components within less-privilege processes to limit attackers' access. Moreover, Linux's \glspl{cgroup} and \emph{namespaces} are the core of container technologies like Docker~\cite{merkel2014docker}, which power a large share of modern cloud computing. \gls{os} system call filtering like Linux Seccomp effectively restricts available system calls and their argument, commonly used in conjunction with the mentioned primitives~\cite{docker-seccomp,kubernetes-seccomp}.

Unfortunately, in the modern world, these mechanisms have been proven insufficient. Incidents such as Heartbleed~\cite{heartbleed} that leak the webserver private key due to a buggy TLS component call for finer-grained isolation of program components. However, applying process-based isolation for such program components requires non-trivial engineering efforts. Moreover, process isolation suffers from significant performance degradation due to \gls{ipc} costs and heavy context switching~\cite{erim,lwc}. At the same time, the rise of confidential computing applications through hardware-assisted \glspl{tee} calls for considering more advanced threat models. \Glspl{tee} technologies place the kernel or hypervisor outside of their \glspl{tcb}. Moreover, side-channel attacks~\cite{zhang_klotski_2020,sgx-cache-attack,sgx-grand-exposure,controlled-channel,morbitzer_extracting_2019,morbitzer_severed_2018,wilke_sevurity_2020} means that the promised confidentiality and integrity of \glspl{tee} are threatened.
Against these attacks, we need to revisit the security role of \glspl{os}, especially when the OS is also included within the application, as in the case of VM-based \glspl{tee} like AMD \gls{sev}~\cite{advanced_micro_device_sev_2022}.

Also, security hardware features are constantly released by hardware vendors. Intel \gls{pku} enables page-granularity memory isolation for in-process domains. Recently, ARM released \gls{mte} and \gls{pa} that open up a large design space for exploration~\cite{pac-it-up,pactight,color-my-world}. Still, using these security features properly is non-trivial since there may be pitfalls when security mechanisms are carelessly designed around them. For instance, recent research discussed the issue of kernel confused deputy through the system call interface~\cite{pku-pitfalls,jenny,cerberus}.
For \glspl{tee} technologies, Iago attacks \cite{checkoway_iago_2013} is a concern that must be addressed.


Amidst these issues, two new research challenges must be addressed for modern \gls{os} security. First, how can the modern hardware security features be properly utilized, or \emph{retrofitted} to implement \gls{os} security mechanisms? Second, how \gls{os} designs must be adapted to tackle security issues when adapting modern hardware features?

My first work, \sepim, tries to answer the first question by retrofitting \gls{pim} technologies to be used as accelerators for confidential computing.
I discussed the benefits, as well as carefully studied the security challenges. Based on the observations, I proposed minimal yet imperative extensions to \gls{pim} to mitigate security issues and enable them as secure accelerators for \glspl{tee}.

My second work, \capacity, attempts to answer both questions. It proposes an access control model for in-process domains inspired by capability systems~\cite{cheri,capsicum,eros,cheri-jni}. \capacity utilizes modern ARM hardware features, \gls{pa} and \gls{mte}, which already exhibit characteristics of capabilities, to turn process references into cryptographic references. \capacity protection coherently overs the life-cycles of protected resources, from the time of import from the file system to its location within memory, accessed through pointers.

Finally, \oblivm, a work-in-progress, answers the second question. It enhances the design of unikernels to provide portable and efficient side-channel defense mechanisms. The design is based on the observation that unikernels have direct and efficient access to privileged hardware features.



\textbf{Outline.}
The remainder of this dissertation is structured as follows. I first cover the preliminaries in \autoref{chap:background}. In the next two chapters, \autoref{chap:sepim} and \autoref{chap:capacity}, I describe \sepim and \capacity. Finally, I briefly review \oblivm, a work-in-progress in \autoref{chap:oblivm}.
