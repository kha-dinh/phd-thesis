\chapter{Introduction}
\pagenumbering{arabic}
\label{sec:intro}

The past century has witnessed rapid technological progress, with computing systems among its most transformative developments.
The advent of the internet revolutionized global information sharing and retrieval.
More recently, advances in artificial intelligence have profoundly influenced diverse fields~\cite{}.
Underpinning these developments are the infrastructures that enable computation, managed fundamentally by the \gls{os}—the mediator between hardware and software.

Every \gls{os} pursues three fundamental objectives.
First, it must provide a set of \glspl{api} that are intuitive for application developers.
This is achieved through \emph{abstractions} that describe high-level resource properties while concealing implementation details.

Second, it must efficiently \emph{manage hardware resources}—including CPU time, memory, and storage—by allocating and reclaiming them as needed.
Disciplines such as file system design and scheduling evolved to fulfill this objective.

Third—and the focus of this thesis—the \gls{os} must ensure the \emph{security} of computation.
This objective became essential with the advent of multiprogramming in the 1960s, when systems began supporting multiple concurrent users and programs.
A malicious program could compromise the confidentiality, integrity, or availability of both the \gls{os} and other applications.
Consequently, the \gls{os} must mediate resource access to uphold these guarantees, even in the presence of potentially hostile users.

\glspl{os} ensure security by enforcing a \emph{protection model}~\cite{lampsonprotection} that specifies access control between \emph{subjects} (e.g., users and processes) and \emph{objects} (e.g., memory, files, and resource handles). More specifically, it defines what operations subjects may perform on each object at any given time.
For example, a simple policy might dictate that users cannot access resources owned by others.
Assuming a system with three users and three files, this policy can be represented using an \emph{access control matrix}~\cite{lampsonprotection}, as shown in \autoref{tab:example-acm}.


\glspl{os} map their \emph{abstractions} into the \emph{subjects} and \emph{objects} of this protection model.
The \emph{process} abstraction, introduced in the 1960s~\cite{denning2022systems}, remains the foundation for representing subjects.
A process is a unit of execution that consumes CPU time and interacts with system resources.
Encapsulating functionality within processes simplifies development—each service can be built and debugged independently—and enables precise resource management.
Commodity \glspl{os}, such as UNIX-based systems, associate users and groups with each process, enforcing protection on a per-process basis.
To represent objects, \glspl{os} employ the \emph{file} abstraction, which initially referred to a block of data on disk.
UNIX-inspired systems generalized this concept so that disk I/O, networking, and interprocess communication all appear as file operations, thereby unifying access under a consistent interface and simplifying access control.

Building on these abstractions, \glspl{os} enforce two primary security boundaries: one between the kernel and \emph{userspace} processes, and another among the processes themselves.
The first boundary, maintained through hardware privilege rings, protects the \gls{os}'s components that enforce security (also called its \gls{tcb}) from being tampered.
The second boundary underpins multiprogramming security, enforced via memory translation and mediation of all userspace resource accesses.

This is the protection model that this thesis refers to as the \emph{legacy protection model}.
Despite substantial advances in hardware and increasingly sophisticated attacks, the conceptual foundation of mainstream \glspl{os} security has remained largely unchanged since the 1970s.
Modern applications depend on long-standing standards \dash\ particularly the \gls{posix} interface \dash which constrains compatibility with alternative architectures.
Consequently, dominant infrastructures continue to rely protection models based on processes and files abstractions, prioritizing \gls{posix} compliance.

Although the legacy protection model has provided a robust foundation for decades, its underlying assumptions no longer align with modern computing environments.
Contemporary software systems are becoming increasingly large and complex, often incorporating untrusted extensions~\cite{}.
Consequently, the coarse-grained process-level protection of traditional \glspl{os} is no longer sufficient containing bugs; the peril which of this is highlighted by the heartbleed attack that exploit a memory corruption error in a cryptographic library.
Furthermore, emerging paradigms such as \emph{confidential computing} invert the traditional trust hierarchy, with the \gls{os} itself no longer considered fully trusted.
This thesis therefore argues that sustaining strong security requires reasoning about the legacy protection models in modern systems.

\section{Misalignment of OS and hardware-defined protection}

Historically, the \gls{os} assumed complete mediation and exclusive control over hardware resources.
However, new hardware components increasingly implement their own protection and isolation mechanisms, often without the \gls{os}’s full visibility or authority.
This shift undermines the \gls{os}’s role as the sole enforcement point for access control.


The \gls{os}, once central to enforcement, is now a potential liability due to misalignment between hardware-enforced and software-enforced protection.

\hdr{Sub-process isolation domains}

As modern applications grow in complexity, features such as Intel \gls{pku} enable sub-process memory isolation beyond traditional page-table mechanisms.
On ARM, Pointer Authentication (PA) and Memory Tagging (MTE) provide pointer integrity and spatial safety.
Research prototypes like \gls{cheri} extend these ideas to full hardware-enforced memory safety and capability-based isolation~\cite{cheri}.
These mechanisms allow components within a single process to coexist under differentiated privileges without incurring the overhead of multiple processes.

However, hardware primitives cannot be applied superficially.
Improper integration may nullify their benefits.
Attacks on in-process isolation mechanisms~\cite{pku-pitfall,schrammel2022jenny,endokernel} demonstrate that existing OS interfaces must be redesigned to safely support sub-process compartments.

\hdr{Trusted execution and CVMs}

Trusted execution technologies significantly alter the protection models assumed in systems.


Taking this principle further, CPUs now support secure “enclaves” that isolate computation even from compromised OSes or hypervisors, enabling remote users to deploy trusted workloads with a minimal \gls{tcb}.

Confidential Virtual Machines (CVMs) encapsulate entire OS instances within hardware-encrypted enclaves.
This approach simplifies deployment by bundling applications with their supporting software.

While CVMs shift trust toward hardware, they inherit a large \gls{tcb} from guest OSes—often millions of lines of code.
They also remain susceptible to side-channel attacks from malicious hypervisors or co-located tenants, as shared resources may amplify leakage through the OS.

\section{Thesis statement}

The divergence between modern hardware capabilities and legacy OS protection abstractions presents both a challenge and an opportunity.
Replacing long-standing OS architectures is infeasible given decades of entrenched software compatibility and standardization.
Yet, ignoring these hardware advances risks leaving core systems unable to leverage the strong, fine-grained protection guarantees that processors now provide.
This thesis argues that the practical path forward lies in \emph{retrofitting}—systematically integrating hardware-backed security mechanisms into existing OS protection models without breaking their foundational abstractions or interfaces.
This has been the approach that has seen the most success.
For example, SELinux~\cite{selinux} extends Linux’s \gls{dac} model with \gls{mac} enforcement, while Capsicum~\cite{watson2010capsicum} retrofits a capability-based \gls{api} in FreeBSD to enable fine-grained access control~\cite{freebsd-capsicum}.
Similarly, large applications such as Chrome and Firefox retrofit finer-grained sub-process isolation to contain vulnerabilities from compromised third-party libraries~\cite{rlbox}.

This thesis asks a fundamental question:

\begin{displayquote}
	\emph{How can operating systems be retrofitted to align with the protection models introduced by modern hardware security mechanisms?}
\end{displayquote}

To answer this, the thesis addresses two key challenges in retrofitting OS security within the evolving hardware-defined landscape:

\begin{itemize}
	\item[-]
	      \emph{How can hardware-based security mechanisms be retrofitted into legacy OS protection models in a principled manner?}

	      Retrofitting must be systematic to avoid misalignments discussed earlier.

	      To address this, I introduce \capacity, a system that retrofits capability-based security into in-process domains~\cite{cheri,capsicum,eros,cheri-jni}.
	      Leveraging ARM features such as Pointer Authentication (\gls{pa}) and Memory Tagging (\gls{mte}), \capacity transforms process references into cryptographically verifiable capabilities.
	      It enforces consistent protection across the resource lifecycle—from file import to memory access—providing hardware-backed integrity and confidentiality guarantees.

This thesis positions itself at the intersection of legacy operating system design and emerging hardware security paradigms.
It argues that while modern processors introduce powerful protection primitives, their security potential can only be realized when operating systems evolve to integrate them coherently.
Through \capacity and \incognitos, this work explores two complementary facets of this evolution: the former demonstrates how in-process security can be retrofitted into existing abstractions, while the latter redefines the OS’s role under hardware-enforced isolation.
Together, they advance the broader goal of rethinking how modern computing infrastructures can achieve principled, hardware-backed security without sacrificing compatibility or usability.
The subsequent chapters elaborate on these systems, detailing their design, implementation, and evaluation.
