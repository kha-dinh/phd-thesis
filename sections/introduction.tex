The past century has witnessed rapid technological progress, with computing systems among its most transformative developments.
The advent of the Internet revolutionized global information exchange. At the same time, recent advances in artificial intelligence have accelerated scientific discovery at an unprecedented pace.
Underlying these breakthroughs are the infrastructures that enable computation \dash\ both software and hardware.
The \gls{os}, which mediates between them, remains a cornerstone of technological innovation.

For decades, the \gls{os} has served as the central authority for enforcing system security, providing a coherent security model that governs users' access to resources.
However, such a model is showing limitations when used to secure modern software.
At the same time, advances in processor design have introduced new hardware-level security mechanisms, such as Intel \gls{pku} and \glspl{sgx}, that enforce protection models partially outside the \gls{os}'s control.
Consequently, system security is now jointly enforced by two distinct entities: the hardware and the software (\gls{os}).
When their respective protection models diverge, gaps in protection may emerge, exposing attack vectors and undermining the intended guarantees of both.

This dissertation argues that hardware-assisted protection models must be systematically coherent with those established by the software (\gls{os}).
Through the lens of coherent protection, it explores two modern challenges in systems security: hardware-assisted in-process isolation and side-channel mitigation in \gls{vm}-level \glspl{tee}.
To address these challenges, the dissertation presents two systems, \capacity and \incognitos, which \emph{retrofit} \gls{os} security to modern hardware security features in a principled and coherent manner.
% These systems provide blueprints for building future systems that leverage the performance and security benefits of modern hardware while maintaining a robust protection model against modern threats.

\section{Motivation}





\subsection{Limitations of existing OS protection}

Historically, the \gls{os} has been the central figure in ensuring systems' security, enforcing a \emph{protection} model~\cite{lampsonprotection}.
This protection governs how programs access system resources, relying on long-standing abstractions such as processes to represent executing principals, and files to represent protected objects.
Combined with hardware support, these abstractions create two fundamental isolation boundaries: one separating user applications from the privileged kernel and another isolating processes from each other.

The traditional, process-centric \emph{protection} has served as the basis of OS security for decades. Still, its assumptions no longer align with modern computing environments.
Two key weaknesses are revealed by attempts to add security to modern systems.

\begin{itemize}
	\item First, the process-level isolation boundaries provided by the OS are no longer sufficient in containing bugs.
	      Contemporary software systems are increasingly large and complex—a peril underscored by the infamous Heartbleed vulnerability~\cite{heartbleed}, which exploited a bug in the OpenSSL cryptographic library.
	      These weaknesses call for the \emph{separation of privilege} within complex programs into smaller, isolated modules~\cite{capacity,erim,hakc,rlbox,privtrans,gudka2015clean}.
	      The coarse-grained, process-level protection of traditional \glspl{os} becomes a bottleneck for this decomposition, imposing significant development and performance costs~\cite{erim,capacity}.
	\item
	      Second, emerging \emph{confidential computing paradigms}~\cite{confidential,google-confidential-computing,sgx,kaplan_amd_nodate,cca} invert traditional trust hierarchies by placing the \gls{os} \emph{outside} of the trust boundary, necessitating a reconsideration of security goals.
	      Confidential computing, leveraging modern hardware features, establishes \emph{secure enclaves} \dash\ containers that are isolated from the rest of the system, even the OS or the hypervisor.
	      Such a paradigm shift introduces an entirely different trust model that existing OS protection was not designed to express.
	      Yet, previous systems retrofit security into this model, introducing attack surfaces such as the Iago attacks \cite{checkoway2013iago}, which exploit system call APIs to take control of secure enclaves.
\end{itemize}


Given these limitations, this dissertation calls for rethinking the legacy OS's protection model to better align with the security and performance goals of modern systems.

\subsection{Rise of hardware-assisted protection}

At the same time, modern hardware increasingly implements its own protection and isolation mechanisms, often beyond the \gls{os} 's complete visibility or authority.
Taking advantage of these new features, a plethora of systems have been developed and deployed, providing models of \emph{hardware-assisted protection}.
This thesis focuses on the two recent developments enabled by hardware-assisted protection: in-process isolation and VM-level confidential computing, or \gls{cvm}.

\begin{itemize}
	\item \textbf{Domain compression.} In-process isolation (\autoref{inprocess-prot}) \emph{compresses} the protection domains inward to create \emph{sub-process isolation domains}, leveraging CPU features like Intel \gls{pku} that enable  userspace processes to establish memory isolation policies.
	      Works like ERIM and Hodor~\cite{erim,hodor} leverage these hardware extensions to create a protected domain that isolates secrets and contains the impact of buggy code, showing significantly better performance than process-level isolation.
	\item \textbf{Domain expansion.} The \gls{cvm} paradigm (\autoref{cvm-prot}) \emph{expands} the protection domain outward from previous process-level secure enclaves to the entire VM.
	      This direction leverages recent advancements in TEE technologies, such as Intel TDX, AMD SEV, or ARM CCA, to transform an entire VM, including applications and the OS, into a protection domain isolated from the untrusted hypervisor.
	      Isolating the confidential application, along with all its supporting code and data, enables companies to deploy cutting-edge workloads, such as LLM inference, into \glspl{tee} with minimal adaptation costs~\cite{yifan2025pipellm}.
\end{itemize}



Modern systems must leverage these new hardware features to enhance security and performance.
Thus, the challenge for this dissertation is to explore principles for securely incorporating these security features.

\section{Approach}

I now describe the approaches used to address the challenge of adopting modern hardware security features to improve system security and performance.

\subsection{Retrofitting for security}

First, this dissertation argues that the practical path to introduce security is \emph{retrofitting} \dash\ integrating security features into existing protection models without breaking compatibility.
This approach also aligns with the goal of modern hardware features to provide compatibility with existing systems.

Despite substantial advances in security architectures and OS composition, the status quo has remained essentially unchanged since the 1970s.
There have been numerous proposals that promised better security, such as capability architectures~\cite{cap-addressing,cambridge-cap}, OSes~\cite{hydra,eros,cambridge-cap}, microkernels~\cite{sel4}, and secure programming languages~\cite{matsakis_rust_2014,caja}.
Unfortunately, these approaches have not seen large-scale adoption.
Modern applications continue to depend on long-standing standards such as \gls{posix}~\cite{posix} and the Windows API~\cite{winapi}, which constrain compatibility with alternative designs.
This status quo leads the community to believe that achieving security is a hopeless quest~\cite{hopelessquest}.
Ultimately, most current computing infrastructures still rely on OS-level process-level protection.


On the other hand, several successful security retrofits have been adopted, demonstrating the approach's efficacy.
For example, SELinux~\cite{selinux} extends Linux's \gls{dac} model with \gls{mac} enforcement, while Capsicum~\cite{watson2010capsicum} retrofits a capability-based \gls{api} in FreeBSD to enable fine-grained access control~\cite{freebsd-capsicum}.
Especially, SELinux is now a core component of Android, running on millions of devices~\cite{droidcap}.
Similarly, large applications such as Chrome and Firefox are retrofitting sub-process isolation to contain vulnerabilities from compromised third-party libraries~\cite{rlbox}.


\subsection{Coherent software/hardware protection}\label{subsec:coherent}

While modern processors provide powerful protection primitives, their full security potential can only be realized when \glspl{os} integrates them carefully.
This thesis proposes the following principle when integrating hardware-assisted security features:

\begin{displayquote}
	\itshape
	\textbf{Principle of coherent protection.} Protection enforced by the software (\gls{os}) must be coherent with the protection established with hardware support, toward a unified protection model
	This means (1) any new hardware-defined protection must carefully consider the one established by software, and (2) the software-enforced protection must align with the threat model of new hardware protection.
\end{displayquote}


Previous research has demonstrated that incoherencies in the protection model lead to insecurity \- systems that leverage modern hardware features in an ad hoc manner often reveal gaps in protections that undermine the benefits provided.
For instance, for hardware-assisted in-process isolation systems, prior work has demonstrated \emph{confused deputy} attacks, arising from the \gls{os} 's lack of awareness of hardware-enforced protection domains~\cite{pku-pitfalls,jenny}.
Similarly, trying to retrofit legacy \glspl{os} into VM-level isolation (e.g., AMD-SEV) can exacerbate side-channel and interface vulnerabilities, as demonstrated by recent attacks against confidential virtual machines~\cite{wesee,heckler,ciphersteal,li2021cipherleaks}.
\cref{chap:capacity} and \cref{chap:incognitos} will go into more detail about the problems caused by inconsistencies in the protection.

The work presented in this thesis, \capacity~\cite{capacity}  and \incognitos~\cite{incognitos}, examines different layers of the protection hierarchy, visiting different modern challenges in system security, to illustrate the broader principle of coherent protection.
These systems demonstrate that, safety and efficiency could be achieved through coherent designs.

\section{Contributions \& Publications}

\subsection{Main Contributions}

This dissertation explores the principle of coherent protection with two complementary approaches:
(1) retrofitting security guarantees provided by hardware to those provided by the \gls{os}, and (2) retrofitting the \gls{os} protection model to reflect the one established by hardware.


\hdr{Retrofitting hardware to incorporate OS protection}
At the intra-process level, \capacity~\cite{capacity} demonstrates how intra-process domains established through modern ARM hardware acceleration can be coherently retrofitted into legacy abstractions, with a novel cryptographic capabilities abstraction.
\capacity retrofits capability-based protection into legacy process abstractions, leveraging ARM Pointer Authentication (\gls{pa}) and Memory Tagging (\gls{mte}) to transform resource references into cryptographically verifiable capabilities.
This design enforces consistent security semantics across file and memory accesses, bridging the gap between software abstractions and hardware enforcement while leveraging modern hardware acceleration.


\hdr{Retrofitting OSes for protection of hardware domains}
At the system level, \incognitos~\cite{incognitos} explores how \glspl{os} can collaborate with hardware-enforced VM-level isolation to enforce coherent threat models.
Specifically, \incognitos redefines the role of the \gls{os} in modern \gls{cvm}- protected systems, treating it as an active component in mitigating \emph{side-channel attacks}, a critical attack vector for \gls{tee}.
By retrofitting \gls{os} capabilities, scheduling, and memory management, it mitigates side-channel leakage while incurring reasonable overheads from protection.


\subsection{Orthogonal Contributions}
Beyond CPU features, which are the primary focus of this thesis, hardware vendors also continue to introduce new acceleration hardware.
A key challenge for modern systems security is how to safely leverage such accelerators amid evolving protection models.
During the timeframe of this thesis, I also led and participated in research addressing this challenge, specifically exploring how \gls{pim} can accelerate emerging confidential computing workloads.

\hdr{Accelerating Hardware Security with Processing-in-memory}

As computing workloads become increasingly data-intensive, \gls{pim} is emerging as a promising paradigm to alleviate the \emph{memory wall problem}~\cite{prime,pim-architecture,samsung-pim}.
SE-PIM~\cite{sepim}, a project I led, is a hardware–software co-design for accelerating confidential data-intensive workloads in the cloud. It examines the attack vectors exposed when PIM is employed for confidential computing and investigates the hardware extensions required to enable secure and practical acceleration.
Building on the foundations of \cite{sepim}, PIM-ORAM, a project in which I contributed as second author, explores how a PIM module with confidential computation capabilities can reduce the overheads of ORAM, a class of security algorithms known for high memory and performance costs when mitigating side channels.
