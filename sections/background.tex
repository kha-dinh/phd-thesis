\glspl{os} enforce security by implementing a system of \emph{protection}~\cite{lampsonprotection} that defines access control policies between \emph{subjects} (e.g., users and processes) and \emph{objects} (e.g., memory, files, and resource handles).
Specifically, it determines which operations each subject may perform on a given object at any time.
For example, a simple policy might prohibit users from accessing resources owned by others.
In a system with three users and three files, this policy can be represented as an \emph{access control matrix}~\cite{lampsonprotection}, as shown in \autoref{tab:example-acm}.

\glspl{os} map their \emph{abstractions} onto the \emph{subjects} and \emph{objects} of this protection.
The \emph{process} abstraction, introduced in the 1960s~\cite{denning2022systems}, remains the foundation for representing subjects.
A process encapsulates relevant functionalities and interacts with system resources.
Commodity \glspl{os}, such as UNIX-based systems, associate processes with users and groups, enforcing protection on a per-process basis.

To represent objects, \glspl{os} employ the \emph{file} abstraction, which initially referred to a block of data on disk.
UNIX-inspired systems generalized this concept so that disk I/O, networking, and system services all appear as file operations, thereby unifying access under a consistent interface and simplifying access control.

Building on these abstractions, the \gls{os} enforces two primary security boundaries: one between the kernel and \emph{userspace} processes, and another among processes themselves.
The first boundary, maintained through hardware privilege rings, protects the \gls{os}'s components that enforce security (its \gls{tcb}) from tampering.
The second boundary underpins multiprogramming security, enforced through memory translation and mediation of all userspace resource accesses.
In terms of protection, two  \emph{protection domains}~\cite{lampsonprotection} are established: user space domains where applications run, and kernel domain where the kernel code executes.
In this thesis, the \emph{legacy protection} refers to the process- and file-centric one that underpins UNIX and POSIX-compatible \gls{os}.

\subsection{Memory corruption}
Memory corruption bugs may be classified into \emph{spatial} - bugs that exploit out-of-bound pointers dereference - and \emph{temporal} - bugs that exploit the dereference of pointers that the memory allocator has freed. Using memory corruption as the primitive, more complex attack, such as control-flow diversion attacks, is performed, as I will show later in this section.

\autoref{fig:bg-spatial-bug} shows a rudimentary example of a spatial memory bug. A case of \emph{buffer overflow} causes the memory corruption in the example. Buffer overflows are commonly caused by performing memory access in an unsafe manner. In the example, the C standard function \mintinline{c}{gets} reads the content from the standard input into a buffer in the argument. However, the function might overwrite memory \emph{outside} of the buffer since no length is specified. Even worse, since \mintinline{c}{gets} reads input from the terminal, an attacker could inject arbitrary value at specified memory locations through carefully crafted \emph{payloads}.

\subsection{Remedies for memory safety issues}
Over the years, industries and researchers have made enormous progress toward solving memory safety issues.
However, it is still a difficult problem to solve.

One approach would be to rewrite legacy software into memory-safe languages like Rust.
However, this is often an immeasurable task due to the vast core software infrastructures written in C and C++, many of which have become irreplaceable.

Two prominent remedies for memory safety issues are \emph{runtime defenses} and software \emph{sanitizer} memory bugs from unsafe programs. Runtime defenses try to mitigate the effect of memory exploits, e.g., control-flow hijacking. For instance, \gls{cfi} mechanisms compute valid call sites control-flow transfer sites, and check whether the control-flow transfer is valid at runtime.
Software sanitizers try to detect memory errors as they happen. However, it comes with substaintial performance overheads~\cite{song_sok_2019,asan}, and is often used to find bugs pre-production.

\section{Trusted Execution Environments}
Modern cloud computing allows users to deploy applications to remote machines hosted by the \gls{csp}.
Traditionally, this deployment model requires the cloud users to fully trust the \gls{csp}.
More specifically, while the size of the cloud applications may be limited, the \gls{tcb} for cloud applications include the operating system (that the \gls{csp} installed) and the machines that the program executes on, now under the control of the \gls{csp}.

However, for modern high-assurance applications, such as private data processing, this model of trust is insufficient.
The operating systems, given their large code base, are often the first target of many software attacks.
With the \gls{os} privilege, attackers could easily obtain and modify the memory content of any running applications to compromise their confidentiality and integrity.
Moreover, with control over the physical machine, a potentially malicious \gls{csp} could launch attacks such as cold boot attacks \cite{halderman_lest_2009} to extract the memory content of executing sensitive applications.

\input{figures/bg-tee-trust-model-tikz}


\Gls{tee} technologies allow remote users to discard trust in the system that runs the sensitive application.
Instead, the \gls{tcb} of the \gls{tee}-protected applications would only include the application and the hardware. \autoref{fig:bg:trust-model-tee} demonstrates the comparison of the trust models of traditional computing and enclave-based \glspl{tee} such as Intel SGX~\cite{costan_intel_2016}.
While traditional computing (left figure) requires that the application owner trusts the entire system that hosts the application, including the \gls{os}, the hypervisor, and the hardware, the \gls{tcb} of \gls{tee}-based applications (right side) is limited to only the \gls{tee}-enabled hardware, and a limited, trusted part of the application called the \emph{enclave} (in the case of Intel SGX).

\subsection{Functionalities of TEEs}
Most \gls{tee} technologies provide memory isolation primitives, such that the memory space of \gls{tee}-protected applications are isolated from each other and the rest of the system. Memory encryption is also a common feature~\cite{advanced_micro_device_sev_2022,costan_intel_2016} to thwart cold boot and memory scanning attacks.

\emph{Remote attestation} is another core functionality of \glspl{tee}. It allows a remote user to verify the state of a deployed program (e.g., initial memory, developer's signature, ...) and the authenticity of a \gls{tee}-enabled CPU.
A simplified procedure for remote attestation is as follows.
To perform remote attestation, the CPU first measures the state of a deployed application and generates a checksum on this data.
The checksum is then cryptographically \emph{signed} using an \emph{attestation key} only known to the CPU that is \emph{provisioned} during manufacture time.
Attestation keys are provisioned in \emph{pairs}.
The manufacturer burns one attestation key into a fuse of the CPU. It keeps the other to verify signatures created the one burnt into the CPU\footnote{Attestation keys are similar to the public and private key of the private key infrastructure}.

The signed checksum is returned to the remote attestation requester (i.e., the \emph{challenger}).
The challenger now sends the signed checksum to a \emph{verifier service}, hosted by the \gls{tee} manufacturer.
This verifier service holds the other, which it can use to verify the signed checksum.
If verification succeeds, the challenger can re-compute the expected checksum and compare it with the one returned by the \gls{tee} application.

\subsection{Isolation primitives}

Processes are given dedicated address spaces and \gls{os} resource handles in a \emph{sandboxed} environment.
However, this abstraction is proven insufficient for modern high-performance applications, due to the high overheads of inter-process communication.

Not all applications fully utilize the full spectrum of system calls the \gls{os} provides.
However, most system calls are available to any application through the standard library, and many remain unused throughout the lifetime of processes. These unused system calls are a form of \emph{over-privilege}, which, when left intact, would grant the attackers additional capabilities in case the application is compromised. For instance, a compression application would unlikely need access to networking system calls such as \texttt{sys\_socket} and \texttt{sys\_listen}. In another case, common applications only operate on user-specified files and do not need access to the entire file system of the user. With such access, there would be catastrophic consequences when the control is taken by attackers, such as leaking private keys or a corrupted file system. Following the Principle of Least Privilege, most modern \glspl{os}, including Linux, OpenBSD, and macOS provides \emph{filtering} mechanisms to specify the allowed system calls a process can invoke.
